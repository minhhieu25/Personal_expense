## high: Wallet balances can be corrupted when transaction writes fail or race

id: fnd_sig-feat-library-592143abcc-a57d_d7fb78d36a
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/models.py:64-109 (Transaction.save)
- Finances/models.py:111-119 (Transaction.delete)

Transaction.save mutates and saves wallet balances before saving the transaction itself, and delete mutates the wallet before deleting the transaction. These operations are not wrapped in a database transaction and do not lock or use atomic F-expression updates. If the transaction save/delete fails after the wallet save, the wallet balance no longer matches persisted transactions. Concurrent creates/edits/deletes against the same wallet can also overwrite each other's balance calculations because each request reads and writes a stale Wallet.balance value in Python.

recommendation:
Wrap transaction and wallet updates in django.db.transaction.atomic(), lock affected wallet and old transaction rows with select_for_update(), and update balances with F expressions or recompute from transactions after commit-sensitive changes.

test analysis:
Finances/tests.py contains only the generated placeholder and no tests for transaction save/delete side effects, failure rollback, or concurrent balance updates.

suggested regression test:
Add tests that simulate a failed Transaction.save after wallet mutation and concurrent transaction creation/editing for the same wallet, asserting the final Wallet.balance equals the sum of persisted transactions.

minimum fix scope:
Transaction.save and Transaction.delete balance mutation logic.

## medium: Invalid form posts can return no response

id: fnd_sig-feat-library-592143abcc-ef6a_5351c6504b
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/views.py:48-53 (CategoryEditView.post)
- Finances/views.py:107-112 (WalletEditView.post)
- Finances/views.py:157-164 (TransactionEditView.post)
- Finances/views.py:221-228 (BudgetEditView.post)

These POST handlers return a redirect only when form.is_valid() is true. If validation fails, the method falls off the end and returns None, which Django treats as an invalid view response and raises an error instead of re-rendering the form with validation messages. Users can trigger this with malformed or incomplete edit submissions.

recommendation:
For each edit POST handler, re-render the corresponding edit template with the bound invalid form and object context when form.is_valid() is false.

test analysis:
Finances/tests.py has no tests for invalid form submissions to edit endpoints.

suggested regression test:
Post invalid payloads to each edit URL as an authenticated user and assert a 200 response containing the bound form errors instead of an exception.

minimum fix scope:
Invalid-form branches in CategoryEditView.post, WalletEditView.post, TransactionEditView.post, and BudgetEditView.post.

## medium: Several personal finance views are reachable without LoginRequiredMixin

id: fnd_sig-feat-library-592143abcc-ab2c_814cd4fd32
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/views.py:39-60 (CategoryEditView, CategoryDeleteView)
- Finances/views.py:98-119 (WalletEditView, WalletDeleteView)
- Finances/views.py:146-171 (TransactionEditView, TransactionDeleteView)
- Finances/views.py:210-235 (BudgetEditView, BudgetDeleteView)
- Finances/views.py:238-327 (CalenderView, SettingView, ReportView)

The list/add views generally use LoginRequiredMixin, but edit/delete and several read pages do not. These views still operate on request.user and personal finance data, so anonymous users can reach the endpoints. Some object views may 404 because they filter by user=request.user, but calendar, settings, and report have no explicit authentication gate, and unauthenticated access can expose pages or trigger errors instead of redirecting to login. This is an inconsistent auth boundary for personal financial functionality.

recommendation:
Add LoginRequiredMixin to every class-based finance view that reads or mutates user-owned data, including edit/delete, calendar, settings, and report views. Keep the existing user=request.user object filters for object-level isolation.

test analysis:
Finances/tests.py has no assertions for anonymous access behavior or redirects to the login page.

suggested regression test:
For each URL in Finances/urls.py, add anonymous-client tests asserting protected finance pages redirect to login, plus authenticated-client tests asserting users cannot access another user's objects.

minimum fix scope:
Finances/views.py class declarations for all unauthenticated finance views.

## medium: Wallet names are globally unique across all users

id: fnd_sig-feat-library-592143abcc-b136_608d9c8784
category: bug
confidence: medium
triage: risk
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/models.py:23-32 (Wallet.Meta)
- Finances/views.py:63-76 (WalletView.get)
- Finances/views.py:85-92 (WalletAddView.post)

Wallet is a user-owned model and the wallet views consistently scope wallet operations to request.user, but the model uniqueness constraint only includes name. That means one user's common wallet name such as 'Cash' or 'Bank' can prevent every other user from creating a wallet with the same name, which breaks per-user isolation and can make normal account setup fail with an IntegrityError or form validation error.

recommendation:
Change the uniqueness constraint to include both user and name, preferably with a UniqueConstraint(fields=['user', 'name'], name='unique_wallet_name_per_user'), and add a migration.

test analysis:
Finances/tests.py has no multi-user model or form tests for wallet creation.

suggested regression test:
Create two users and assert both can create a wallet with the same name, while a single user cannot create duplicate wallet names.

minimum fix scope:
Wallet.Meta uniqueness definition and the corresponding migration.
