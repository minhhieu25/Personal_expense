## high: Editing an existing transaction corrupts wallet balance when the wallet is unchanged

id: fnd_sig-feat-library-592143abcc-9b15_fa013cfc9f
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/models.py:63-91 (Transaction.save)

For existing transactions, the old amount is reversed on old.wallet, but old.wallet is only saved when the wallet changes. When the wallet stays the same, that reversal is discarded because self.wallet is a separate loaded instance; the new amount is then applied on top of the pre-edit balance. Editing an income transaction from 100 to 50 on the same wallet turns a 100 balance into 150 instead of 50, and expense edits have the equivalent error.

recommendation:
Apply the delta when the wallet is unchanged, or reload and update a single wallet instance inside an atomic transaction. Save both old and new wallets only when they are different.

test analysis:
Finances/tests.py contains only the default TestCase import and no transaction balance tests.

suggested regression test:
Add a test that creates income and expense transactions, edits amount/type/category with the same wallet, and asserts the wallet balance equals the recomputed transaction total.

minimum fix scope:
Update Transaction.save balance adjustment logic and cover same-wallet edit cases.

repro:
Create a wallet with balance 0, create a 'Thu nhập' transaction for 100, then edit the same transaction to amount 50 without changing wallet. The wallet balance becomes 150 instead of 50.

## high: Production-unsafe Django secrets and debug mode are hard-coded

id: fnd_sig-feat-library-dce6924bb2-f0e2_8a90a6981c
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Python source MySite (feat_library_dce6924bb2)

evidence:
- MySite/settings.py:23-29
- MySite/settings.py:79-87

The settings module hard-codes the Django SECRET_KEY and PostgreSQL credentials while also enabling DEBUG and leaving ALLOWED_HOSTS empty. If this settings module is deployed or committed/shared as-is, disclosure of the secret key can allow session/signing forgery risks, database credentials are exposed, and DEBUG can leak sensitive runtime details. The ASGI/WSGI entrypoints always load this same settings module, so there is no separate production-safe configuration path in the owned files.

recommendation:
Read SECRET_KEY, DEBUG, ALLOWED_HOSTS, and database credentials from environment variables or a deployment secret store, default DEBUG to false for non-local environments, and fail startup when required production settings are missing.

test analysis:
No tests are included for settings validation or production deployment safety.

suggested regression test:
Add a settings smoke test that imports the production settings environment with DEBUG disabled and asserts secrets/DB credentials are not literal defaults and required host configuration is present.

minimum fix scope:
Update MySite/settings.py to externalize secrets and environment-specific deployment settings.

## medium: Invalid edit form submissions return no HTTP response

id: fnd_sig-feat-library-592143abcc-9168_73a48d57c2
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/views.py:48-54 (CategoryEditView.post)
- Finances/views.py:107-113 (WalletEditView.post)
- Finances/views.py:157-165 (TransactionEditView.post)
- Finances/views.py:221-229 (BudgetEditView.post)

Each edit post handler redirects only when form.is_valid() is true and otherwise falls off the end of the method. Django class-based view handlers must return an HttpResponse; invalid input or uniqueness violations will raise a server error instead of redisplaying the form with validation errors.

recommendation:
For every edit post handler, render the edit template with the bound invalid form and object context, preserving filtered querysets where applicable.

test analysis:
Finances/tests.py has no tests for invalid form submissions.

suggested regression test:
Add tests posting invalid data to category, wallet, transaction, and budget edit URLs and assert a 200 response with form errors.

minimum fix scope:
Add invalid-form return paths to the affected edit views.

repro:
Submit an edit form with invalid data, such as a non-numeric amount for a wallet or duplicate values that violate model validation. The post method returns None.

## medium: Logout endpoint accepts CSRFable GET requests

id: fnd_sig-feat-library-a5f9463f7f-5b37_b71121d571
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Python source Users (feat_library_a5f9463f7f)

evidence:
- Users/urls.py:7-8 (urlpatterns)
- Users/views.py:60-63 (LogoutView.get)

The logout route is exposed as a normal URL and LogoutView performs logout in response to GET. Because GET requests do not require a CSRF token and can be triggered cross-site by links, images, or redirects, another site can force an authenticated user to be logged out. This is a state-changing action and should require POST with CSRF protection.

recommendation:
Require POST for logout, use Django's CSRF protection, and update templates to submit a POST form or use Django's built-in LogoutView with POST-only behavior.

test analysis:
Users/tests.py contains no tests, so there is no assertion that logout rejects GET or requires CSRF-protected POST.

suggested regression test:
Add a test that GET /logout/ does not log out the user and that POST /logout/ logs out only through the intended route.

minimum fix scope:
Change Users.views.LogoutView to handle logout via POST only and update Users.urls/templates or callers accordingly.

repro:
While logged in, visit or embed a request to the logout URL from any page; the server will execute LogoutView.get and clear the session.

## medium: Unauthenticated users can hit finance views that assume an authenticated user

id: fnd_sig-feat-library-592143abcc-c39d_26bfed42f9
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/views.py:39-60 (CategoryEditView, CategoryDeleteView)
- Finances/views.py:98-119 (WalletEditView, WalletDeleteView)
- Finances/views.py:146-171 (TransactionEditView, TransactionDeleteView)
- Finances/views.py:210-235 (BudgetEditView, BudgetDeleteView)
- Finances/views.py:238-327 (CalenderView, SettingView, ReportView)

Several finance views omit LoginRequiredMixin while reading request.user or exposing finance-related pages. Unlike the list/add views, unauthenticated requests are not redirected to login; views that query user-owned models may error with AnonymousUser, and settings/report/calendar pages are reachable without the same authentication gate as the rest of the finance feature.

recommendation:
Add LoginRequiredMixin to every finance view that displays or mutates user finance data, including edit/delete, calendar, settings, and report views.

test analysis:
There are no access-control tests in Finances/tests.py.

suggested regression test:
Add tests asserting anonymous GET/POST requests to every finance URL redirect to login, and authenticated users can only access objects they own.

minimum fix scope:
Add LoginRequiredMixin consistently across the affected class-based views.

repro:
As an anonymous user, request routes such as /calendar/ or /transaction/1/edit/. These classes do not enforce login before executing their handlers.

## medium: Wallet name uniqueness is global instead of per user

id: fnd_sig-feat-library-592143abcc-b475_0fbb2e6f9a
category: bug
confidence: medium
triage: risk
status: open
feature: Python source Finances (feat_library_592143abcc)

evidence:
- Finances/models.py:23-32 (Wallet.Meta)
- Finances/views.py:85-95 (WalletAddView.post)

Wallets are user-owned, and all wallet queries are scoped to request.user, but the model uniqueness constraint only includes name. This prevents different users from creating wallets with the same common name, such as 'Cash' or 'Bank', even though their wallets are otherwise isolated by user.

recommendation:
Change the wallet uniqueness constraint to include both user and name, and add a migration. Prefer a modern UniqueConstraint if supported by the Django version in use.

test analysis:
Finances/tests.py has no wallet creation or multi-user uniqueness tests.

suggested regression test:
Add a test showing two different users can create wallets with the same name, while one user cannot create duplicate wallet names.

minimum fix scope:
Update Wallet.Meta uniqueness and create the corresponding migration.

repro:
User A creates a wallet named 'Cash'. User B then submits the wallet add form with name 'Cash'; the model uniqueness constraint rejects it.

## low: Application route lacks a trailing slash

id: fnd_sig-feat-library-dce6924bb2-af80_95aa67e005
category: api-contract
confidence: medium
triage: contract-mismatch
status: open
feature: Python source MySite (feat_library_dce6924bb2)

evidence:
- MySite/urls.py:22-29

The project routes the Finances app at 'finances' instead of the conventional directory-style 'finances/'. With Django's CommonMiddleware and APPEND_SLASH behavior, requests to '/finances/' will not be redirected to '/finances' and can 404 depending on the included URL patterns. This creates an avoidable route contract mismatch for users and links that naturally include the slash.

recommendation:
Change the include mount to path('finances/', include('Finances.urls')) and update any internal links that target the no-slash path.

test analysis:
No URL routing tests are included for the project-level URL configuration.

suggested regression test:
Add a URL resolver or client test asserting that '/finances/' resolves or returns the expected Finances app response.

minimum fix scope:
Update the Finances route in MySite/urls.py and any callers that depend on '/finances' exactly.

repro:
Request /finances/ against this URL configuration when the included Finances.urls patterns are rooted normally; Django will not match the parent prefix 'finances' and the slash-appending middleware cannot fix an already-slashed URL.
